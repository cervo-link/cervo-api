# cervo-api

Bookmark management API with vector search, built to power chat platform integrations (Discord, Slack). Users save URLs; the API scrapes, summarizes, and embeds them — then surfaces the most relevant ones through natural language search.

## Overview

**What it does:**

1. A user submits a URL via the API (or through a Discord/Slack bot)
2. The API immediately returns a bookmark with `status: submitted`
3. In the background, it scrapes the page content, generates a summary, title, and tags via a local LLM (Gemma), and creates a 768-dimension vector embedding
4. Users search their workspace bookmarks using natural language; the query is embedded and matched via cosine similarity (`pgvector`)
5. Search results include an AI-generated explanation of why each bookmark matched

**Stack:** Fastify · Drizzle ORM · PostgreSQL + pgvector · Ollama/Gemma (local LLM) · ScrapingBee · Bun

## Architecture

Clean Architecture — domain logic has zero framework or database dependencies.

```
src/
├── domain/
│   ├── entities/          # TypeScript types inferred from Drizzle schema
│   ├── services/          # Pure business logic, injected adapters
│   └── errors/            # Domain error classes with HTTP status codes
│
├── infra/
│   ├── adapters/
│   │   ├── gemma/         # Ollama/Gemma: summarize, title, tags, explain
│   │   ├── embeddinggemma/# Ollama/Gemma: vector embeddings (768 dims)
│   │   ├── scrapping-bee/ # ScrapingBee + X/Twitter oEmbed scraper
│   │   ├── email/         # Nodemailer (prod) or console logger (dev)
│   │   └── x/             # X/Twitter oEmbed content extraction
│   ├── db/
│   │   ├── schema/        # Drizzle table definitions (single source of truth)
│   │   ├── repositories/  # Data access — one file per table
│   │   └── migrations/    # SQL migration files (generated by Drizzle)
│   ├── http/
│   │   ├── controllers/   # One controller per domain area
│   │   ├── routes/        # Route registration + schema attachment
│   │   ├── schemas/       # Zod schemas for request/response validation
│   │   ├── middlewares/   # apiKeyAuth, jwtAuth
│   │   └── app.ts         # Fastify setup, Swagger UI, error handler
│   ├── factories/         # Instantiate adapters by name string
│   └── ports/             # TypeScript interfaces for all adapters
│
├── tests/
│   ├── global-setup.ts    # Spins up Testcontainer PostgreSQL + runs migrations
│   └── factories/         # Test data helpers (makeMember, makeBookmark, etc.)
│
├── config.ts              # All env vars validated with Zod
└── main.ts                # Entry point
```

### Dependency flow

```
routes → controllers → factories → services → repositories / adapters
```

- Services accept adapter interfaces (ports), not concrete implementations
- Factories resolve provider names to concrete adapters
- Domain errors are return values, never thrown (no `try/catch` in callers)
- `@/*` path alias maps to `src/*`

## Getting Started

### Prerequisites

- [Bun](https://bun.sh) >= 1.0
- [Docker](https://docker.com) (for PostgreSQL + OTEL stack)
- [Ollama](https://ollama.ai) with `gemma:2b` pulled (for summarization/embedding)
- [ScrapingBee](https://www.scrapingbee.com) API key (for scraping)

### Setup

```bash
# Install dependencies
bun install

# Start infrastructure (PostgreSQL + Grafana OTEL LGTM)
docker compose up -d

# Copy and fill in environment variables
cp .env.example .env

# Run migrations
bun run db:migrate

# Start dev server with hot-reload
bun run dev
```

Swagger UI is available at `http://localhost:8080/swagger` in development.

### Commands

```bash
# Development
bun run dev              # Start with hot-reload
bun run build            # TypeScript compilation

# Database
bun run db:generate      # Generate Drizzle migrations from schema changes
bun run db:migrate       # Apply migrations to DATABASE_URL
bun run db:migrate:test  # Apply migrations to test database

# Testing
bun run test             # Run all tests (requires Docker for Testcontainers)
bun run test:watch       # Watch mode
bun run test:coverage    # Coverage report

# Code quality
bun run typecheck        # tsc --noEmit
bun run lint             # Biome lint check
bun run lint:fix         # Biome lint auto-fix
bun run format           # Biome format
```

Run a single test file:
```bash
bun --env-file=.env.test run vitest src/domain/services/bookmarks/get-bookmark-service.spec.ts
```

```
  id            UUID PK

  id            UUID PK

  createdAt     TIMESTAMP
  updatedAt     TIMESTAMP
  active        BOOLEAN DEFAULT true

member_platform_identities           -- links member ↔ Discord user, Slack user, etc.
  id            UUID PK
  memberId      UUID → members.id
  provider      TEXT NOT NULL        -- e.g. "discord"
  providerUserId TEXT NOT NULL       -- platform-specific user ID
  createdAt     TIMESTAMP
  UNIQUE (provider, providerUserId)

  id            UUID PK
  workspaceId   UUID → workspaces.id
  memberId      UUID → members.id
  url           TEXT NOT NULL
  urlHashId     CHAR(64) NOT NULL    -- SHA-256 of URL for deduplication
  status        TEXT DEFAULT 'submitted'  -- submitted | ready | failed
  title         TEXT                 -- LLM-generated
  description   TEXT                 -- LLM-generated summary
  tags          TEXT[]               -- 3-5 LLM-generated tags
  failureReason TEXT
  embedding     VECTOR(768)          -- pgvector; null until processing completes
  createdAt     TIMESTAMP
  updatedAt     TIMESTAMP
  visible       BOOLEAN DEFAULT true

magic_link_tokens
  id            UUID PK
  memberId      UUID → members.id
  token         TEXT UNIQUE
  expiresAt     TIMESTAMP
  usedAt        TIMESTAMP            -- set when consumed
  createdAt     TIMESTAMP

refresh_tokens
  id            UUID PK
  memberId      UUID → members.id
  token         TEXT UNIQUE
  expiresAt     TIMESTAMP
  revokedAt     TIMESTAMP            -- set on logout
  createdAt     TIMESTAMP
```

## Async Processing Pipeline

When a bookmark is created, the HTTP response returns immediately with `status: submitted`. Processing runs in the background via `setImmediate`:

```
[submitted]
    ↓
scrape URL content (ScrappingBee / X oEmbed)
    ↓ fail → status: failed, failureReason set
summarize content → description (Gemma)
    ↓ fail → status: failed, failureReason set
generate title (Gemma)             ← non-critical: failure is silently ignored
generate tags (Gemma)              ← non-critical: failure is silently ignored
embed description (Ollama)
    ↓ fail → status: failed, failureReason set
[ready]
```

**Critical steps** (scrape, summarize, embed): failure sets `status: failed` with a `failureReason`.
**Non-critical steps** (title, tags): failure is silently ignored; the bookmark still becomes `ready`.

Failed bookmarks can be retried via `POST /bookmarks/:id/retry`. This resets `status` to `submitted`, clears `failureReason`, and re-triggers the pipeline.

**X/Twitter URLs** use a custom oEmbed scraper instead of ScrapingBee.

## Authentication

Two mechanisms are used for different purposes:

### API Key (machine-to-machine)

Intended for bot integrations (Discord, Slack) and server-to-server calls. Required on all routes.

```
Authorization: Bearer <api-key>
X-API-Key: <api-key>
?api_key=<api-key>
```

### Magic Link + JWT (user-facing)

Passwordless authentication for human users:

1. `POST /auth/magic-link` → server emails a one-time link
2. User clicks link → `POST /auth/verify` → receives `accessToken` + `refreshToken`
3. `accessToken` — short-lived JWT (default 15 min) for user-scoped operations
4. `refreshToken` — long-lived JWT (default 7d) to obtain new access tokens
5. `POST /auth/logout` — revokes the refresh token

On first sign-in, a default personal workspace is automatically created for the new member.

Passwords are hashed with argon2. JWT tokens are signed with JOSE.

## Testing

Tests use **Testcontainers** to spin up a real PostgreSQL instance with the pgvector extension — no database mocking. The container is shared across the entire test run and torn down at the end.

External adapters (ScrapingBee, Gemma, embedding) are mocked with `vi.fn()` in all tests.

```bash
# Run all tests (Docker must be running)
bun run test

# Watch mode
bun run test:watch

# Single file
bun --env-file=.env.test run vitest src/path/to/file.spec.ts

# Coverage
bun run test:coverage
```

**Test factories** in `src/tests/factories/` create real database records:

```typescript
const member = await makeMember()
const workspace = await makeWorkspace()
await makeMembership(workspace.id, member.id)
const bookmark = await makeBookmark({ workspaceId: workspace.id, memberId: member.id })
```

**End-of-phase check order:** `test → typecheck → lint:fix → lint`

## Key Decisions

### Fire-and-forget bookmark processing

Bookmark creation returns `201` immediately with `{ id, status: "submitted" }`. The scrape/summarize/embed pipeline runs asynchronously via `setImmediate`. This keeps response times fast regardless of LLM or scraping latency. Clients can poll the bookmark status or rely on webhooks.

### Domain errors as return values

Services never throw for expected failures. Instead they return `DomainError | T`. Command-style services (no meaningful success value) return `DomainError | null`. This makes error handling explicit at every call site and eliminates unexpected exceptions propagating up the stack.

### Platform decoupling via identity tables

`workspace_integrations` and `member_platform_identities` decouple domain entities from platform identities (Discord server ID, Slack user ID). A workspace can be linked to multiple platforms; a member can have multiple platform identities. The bot looks up the workspace/member by platform ID without any platform-specific columns on the core tables — adding a new platform requires zero schema changes.

### Testcontainers over mocked DB

Mocked databases let subtle bugs through (wrong column names, constraint violations, query logic errors). All tests hit a real PostgreSQL + pgvector instance. The only mocks are external HTTP services (ScrapingBee, Ollama).

### Auto-membership on workspace creation

When a workspace is created, the owner is automatically added as a member. This is enforced in `createWorkspace` (not scattered across callers) so all workspace creation paths behave consistently.

### Non-critical LLM steps

Title and tag generation are treated as non-critical: if they fail, the bookmark still reaches `ready` with `title: null` and `tags: null`. Only scraping, summarization (required for vector search input), and embedding (required for vector search itself) are critical failures.

### Local LLM via Ollama

All AI features use Gemma 2b running locally via Ollama — no external AI API keys or costs. The embedding dimension is fixed at 768 to match the model output.

### Vector search with explanations

`GET /bookmarks` embeds the search query, finds the nearest bookmarks by cosine distance (`<->` operator in pgvector), then calls the LLM to generate a one-sentence explanation per result (`matchedBecause`). If the explanation step fails, results are still returned — `matchedBecause` is gracefully omitted.
