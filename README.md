# cervo-api

Bookmark management API with vector search, built to power chat platform integrations (Discord, Slack) and the Cervo web app. Users save URLs; the API scrapes, summarizes, and embeds them — then surfaces the most relevant ones through natural language search.

## Overview

**What it does:**

1. A user submits a URL via the API (or through a Discord bot / the web app)
2. The API immediately returns a bookmark with `status: submitted`
3. In the background, it scrapes the page content, generates a summary, title, and tags via OpenAI (gpt-4o-mini), and creates a 1536-dimension vector embedding (text-embedding-3-small)
4. Users search their workspace bookmarks using natural language; the query is embedded and matched via cosine similarity (`pgvector`) with a relevance threshold
5. Search results include an AI-generated explanation of why each bookmark matched

**Stack:** Fastify · Drizzle ORM · PostgreSQL + pgvector · OpenAI · Firecrawl / ScrapingBee · Bun

## Architecture

Clean Architecture — domain logic has zero framework or database dependencies.

```
src/
├── domain/
│   ├── entities/          # TypeScript types inferred from Drizzle schema
│   ├── services/          # Pure business logic, injected adapters
│   └── errors/            # Domain error classes with HTTP status codes
│
├── infra/
│   ├── adapters/
│   │   ├── openai/        # OpenAI: summarize, title, tags, explain, embeddings
│   │   ├── gemma/         # Ollama/Gemma: summarize, title, tags, explain (local fallback)
│   │   ├── embeddinggemma/# Ollama/Gemma: vector embeddings (local fallback)
│   │   ├── firecrawl/     # Firecrawl web scraper
│   │   ├── scrapping-bee/ # ScrapingBee web scraper
│   │   └── x/             # X/Twitter oEmbed content extraction
│   ├── db/
│   │   ├── schema/        # Drizzle table definitions (single source of truth)
│   │   ├── repositories/  # Data access — one file per table
│   │   └── migrations/    # SQL migration files (generated by Drizzle)
│   ├── factories/         # Instantiate adapters by provider name string
│   ├── http/
│   │   ├── controllers/   # One controller per domain area
│   │   ├── routes/        # Route registration + schema attachment
│   │   ├── schemas/       # Zod schemas for request/response validation
│   │   ├── middlewares/   # apiKeyAuth
│   │   └── app.ts         # Fastify setup, Swagger UI, error handler
│   └── ports/             # TypeScript interfaces for all adapters
│
├── tests/
│   ├── global-setup.ts    # Spins up Testcontainer PostgreSQL + runs migrations
│   └── factories/         # Test data helpers (makeMember, makeBookmark, etc.)
│
├── config.ts              # All env vars validated with Zod
└── main.ts                # Entry point
```

### Dependency flow

```
routes → controllers → factories → services → repositories / adapters
```

- Services accept adapter interfaces (ports), not concrete implementations
- Factories resolve provider names (`SCRAPPING_PROVIDER`, `EMBEDDING_PROVIDER`, `SUMMARIZE_PROVIDER`) to concrete adapters
- Domain errors are return values, never thrown (no `try/catch` in callers)
- `@/*` path alias maps to `src/*`

## Getting Started

### Prerequisites

- [Bun](https://bun.sh) >= 1.0
- [Docker](https://docker.com) (for PostgreSQL + OTEL stack)
- [OpenAI API key](https://platform.openai.com) (for summarization and embeddings)
- [Firecrawl](https://firecrawl.dev) or [ScrapingBee](https://www.scrapingbee.com) API key (for scraping)

### Setup

```bash
# Install dependencies
bun install

# Start infrastructure (PostgreSQL + Grafana OTEL LGTM)
docker compose up -d

# Copy and fill in environment variables
cp .env.example .env

# Run migrations
bun run db:migrate

# Start dev server with hot-reload
bun run dev
```

Swagger UI is available at `http://localhost:8090/swagger` in development.

### Commands

```bash
# Development
bun run dev              # Start with hot-reload
bun run build            # TypeScript compilation

# Database
bun run db:generate      # Generate Drizzle migrations from schema changes
bun run db:migrate       # Apply migrations to DATABASE_URL
bun run db:migrate:test  # Apply migrations to test database

# Testing
bun run test             # Run all tests (requires Docker for Testcontainers)
bun run test:watch       # Watch mode
bun run test:coverage    # Coverage report

# Code quality
bun run typecheck        # tsc --noEmit
bun run lint             # Biome lint check
bun run lint:fix         # Biome lint auto-fix
```

Run a single test file:
```bash
bun --env-file=.env.test run vitest src/domain/services/bookmarks/get-bookmark-service.spec.ts
```

### Bookmarks

**`POST /bookmarks`** — Submit a URL for processing

```json
// Request
{
  "workspaceId": "uuid",
  "memberId": "uuid",
  "url": "https://example.com/article",
  "source": "web"   // optional: "web" | "discord" | "raycast" | "api" (default: "web")
}

// 201 — returns immediately, processing happens in background
{ "id": "uuid", "status": "submitted" }

// 404 — workspace or member not found
{ "message": "Workspace not found" }
```

**`GET /bookmarks`** — Search bookmarks by natural language

| Param | Type | Required | Description |
|---|---|---|---|
| `workspaceId` | UUID | Yes | |
| `memberId` | UUID | Yes | |
| `text` | string | Yes | Natural language search query |
| `limit` | number | No | 1–50, default 5 |

Only returns results with cosine distance < 0.7 (semantically relevant matches). Returns an empty array if nothing is relevant.

```json
// 200
[
  {
    "id": "uuid",
    "workspaceId": "uuid",
    "memberId": "uuid",
    "url": "https://example.com",
    "urlHashId": "abc123...",
    "status": "ready",
    "title": "Example Article",
    "description": "A summary of the article content.",
    "tags": ["typescript", "api", "rest"],
    "source": "web",
    "failureReason": null,
    "visible": true,
    "createdAt": "2024-01-01T00:00:00.000Z",
    "updatedAt": "2024-01-01T00:00:00.000Z",
    "matchedBecause": "This article covers TypeScript API design patterns relevant to your query."
  }
]
```

**`POST /bookmarks/:id/retry`** — Retry a failed bookmark

```json
// 200
{ "message": "Retry triggered" }

// 404 — bookmark not found
{ "message": "Bookmark not found" }

// 409 — bookmark is not in failed state
{ "message": "Bookmark is not in failed state" }
```

## Database Schema

```
bookmarks
  id            UUID PK
  workspaceId   UUID → workspaces.id
  memberId      UUID → members.id
  url           TEXT NOT NULL
  urlHashId     CHAR(64) NOT NULL    -- SHA-256 of URL
  status        TEXT DEFAULT 'submitted'  -- submitted | ready | failed
  title         TEXT                 -- LLM-generated
  description   TEXT                 -- LLM-generated summary
  tags          TEXT[]               -- LLM-generated tags
  failureReason TEXT
  source        TEXT DEFAULT 'web'   -- web | discord | raycast | api
  embedding     VECTOR(1536)         -- pgvector; null until processing completes
  createdAt     TIMESTAMP
  updatedAt     TIMESTAMP
  visible       BOOLEAN DEFAULT true

members
  id            UUID PK
  name          TEXT
  username      TEXT UNIQUE
  email         TEXT UNIQUE
  createdAt     TIMESTAMP
  updatedAt     TIMESTAMP
  active        BOOLEAN DEFAULT true

workspaces
  id            UUID PK
  name          TEXT NOT NULL
  description   TEXT
  isPublic      BOOLEAN DEFAULT false
  createdAt     TIMESTAMP
  updatedAt     TIMESTAMP
  active        BOOLEAN DEFAULT true

memberships
  id            UUID PK
  memberId      UUID → members.id
  workspaceId   UUID → workspaces.id
  createdAt     TIMESTAMP
  UNIQUE (memberId, workspaceId)

workspace_integrations
  id            UUID PK
  workspaceId   UUID → workspaces.id
  provider      TEXT NOT NULL        -- e.g. "discord"
  providerId    TEXT NOT NULL
  createdAt     TIMESTAMP
  updatedAt     TIMESTAMP
  active        BOOLEAN DEFAULT true
  UNIQUE (provider, providerId)

member_platform_identities
  id            UUID PK
  memberId      UUID → members.id
  provider      TEXT NOT NULL
  providerUserId TEXT NOT NULL
  createdAt     TIMESTAMP
  UNIQUE (provider, providerUserId)
```

## Async Processing Pipeline

When a bookmark is created, the HTTP response returns immediately with `status: submitted`. Processing runs in the background via `setImmediate`:

```
[submitted]
    ↓
scrape URL content (Firecrawl or ScrapingBee; X/Twitter uses oEmbed)
    ↓ fail → status: failed, failureReason set
summarize content → description (OpenAI gpt-4o-mini)
    ↓ fail → status: failed, failureReason set
generate title (OpenAI)             ← non-critical: failure silently ignored
generate tags (OpenAI)              ← non-critical: failure silently ignored
embed description (OpenAI text-embedding-3-small, 1536 dims)
    ↓ fail → status: failed, failureReason set
[ready]
```

**Critical steps** (scrape, summarize, embed): failure sets `status: failed` with a `failureReason`.
**Non-critical steps** (title, tags): failure is silently ignored; bookmark still becomes `ready`.

Failed bookmarks can be retried via `POST /bookmarks/:id/retry`.

## Key Decisions

### OpenAI for AI features

All AI features use OpenAI: `gpt-4o-mini` for summarization/title/tags/explain and `text-embedding-3-small` for 1536-dimension embeddings. Local Ollama adapters (Gemma) remain available and can be switched via `EMBEDDING_PROVIDER` and `SUMMARIZE_PROVIDER` env vars.

### Relevance threshold on vector search

`GET /bookmarks` filters results to cosine distance < 0.7, so searches return an empty array rather than the closest-but-irrelevant result. Cosine distance (`<=>`) is used over L2 (`<->`) as it's more appropriate for normalized OpenAI embeddings.

### Configurable providers

Scraping, embedding, and summarization providers are resolved at runtime from env vars via factory functions. Switching from Firecrawl to ScrapingBee (or OpenAI to Ollama) requires only a config change, no code changes.

### Bookmark source tracking

Every bookmark records where it was saved from (`source`: `web` | `discord` | `raycast` | `api`). This is set by the caller at creation time and surfaced in the UI.

### Fire-and-forget processing

Bookmark creation returns `201` immediately. The scrape/summarize/embed pipeline runs asynchronously via `setImmediate`, keeping response times fast regardless of LLM or scraping latency.

### Domain errors as return values

Services never throw for expected failures. They return `DomainError | T`. This makes error handling explicit at every call site.

### Testcontainers over mocked DB

All tests hit a real PostgreSQL + pgvector instance. The only mocks are external HTTP services.
